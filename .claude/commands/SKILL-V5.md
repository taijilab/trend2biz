# YC 视角开源投资分析 Skill（v5 — Decision Engine）

## 核心定位

从 v3「结构化投研报告生成器」升级为：

> **开源投资决策引擎**：不只是"描述项目"，而是"给出判断"——该不该投、什么阶段进入、需要什么前置条件、如何对冲风险。

v5 三大升级：
1. **三层评分系统**（Health → Commercial → Structural）替代原七维平面打分，形成递进式判断链
2. **Maintainer Risk Matrix**：从简单的 bus factor 扩展为六维风险矩阵
3. **Investment Thesis First**：报告第一段就给出核心判断，倒金字塔结构

---

## 一、触发条件

当用户出现以下需求时启用：
- "这个开源项目是否值得投资？"
- "帮我判断这个项目的商业潜力"
- "给我一个 YC 风格的投资判断"
- "是否值得战略合作 / 生态整合？"
- 提供 GitHub 仓库链接要求商业分析
- "investment analysis / VC analysis / due diligence"

---

## 二、报告输出标准

**必须产出**：
1. **Markdown 报告** → `/mnt/user-data/outputs/{项目名}_投资分析.md`
2. **HTML 报告**（如用户要求或适合渲染）→ `/mnt/user-data/outputs/{项目名}_投资分析.html`

**两端一致性**：HTML 与 Markdown 信息、数据、评分必须完全一致。

## 三、工作流程

### Step 0：基础准备

确认输入：
- GitHub URL 是否有效
- 仓库是否存在、是否公开

### Step 1：信息收集（必须先搜后写，绝不跳过）

⚠️ **铁律：先收集全部数据，再写一个字的分析。数据失真 = 报告作废。**

按顺序执行，总计 10-25 次 web_search / web_fetch（组织类仓库需更多）：

**1a. GitHub 仓库（必须 web_fetch 实际页面）**
- `web_fetch` GitHub 主页 → 精确提取：stars / forks / contributors / commits / open issues / open PRs / 许可证 / 主语言
- ⚠️ **必须用页面实际数据，绝不使用缓存/记忆中的数据**
- 仔细阅读 README 全文，理解项目**真正做什么**
- **判断仓库所有者类型**：`github.com/{owner}/{repo}` 的 `{owner}` 是个人还是组织？

**1b. 公司/组织背景调查（如果仓库属于组织，必须执行）**

判断依据：GitHub 页面显示 Organization 标识

**1b-i. GitHub 组织页面**
- `web_fetch` `https://github.com/{org}` → 组织简介、官网链接、公开成员数、公开仓库数、Verified 域名、地区
- 浏览 Popular repositories → 该项目在组织中是否 stars 最多？其他仓库是什么？
- 记录：总仓库数、top 5 仓库及 stars、是否 Verified

**1b-ii. 公司官网**
- 从 GitHub 组织页面提取官网链接
- `web_fetch` 官网首页 → 公司定位 tagline、产品列表
- `web_fetch` /about 或 /team → 创始人、团队、使命
- `web_fetch` /pricing → 商业模式、定价结构
- `web_fetch` /blog 或 /news → 近期公告、产品发布

**1b-iii. 公司基本信息**
- `web_search` "{公司名} company founded headquarters" → 成立时间、注册地、法人结构
- `web_search` "{公司名} team size employees 2025 2026" → 团队规模、增长趋势

**1b-iv. 融资与估值**
- `web_search` "{公司名} funding valuation investors crunchbase" → 每轮融资
- `web_search` "{公司名} latest funding round 2025 2026" → 最新一轮
- 记录完整融资历史表（轮次 / 时间 / 金额 / 领投方 / 估值 / **来源链接**）
- ⚠️ **每条融资记录必须尽量附来源 URL，无来源显示 `—`**

**1b-v. 收入与客户**
- `web_search` "{公司名} revenue ARR customers 2025 2026" → 收入规模、增速、客户数
- 如有该项目的独立收入数据，单独记录

**1b-vi. 战略动态**
- `web_search` "{公司名} acquisition partnership 2025 2026" → 近期收购、合作
- `web_search` "{公司名} IPO lawsuit controversy" → 上市计划、法律风险
- ⚠️ **战略动态必须清洗为结构化列表**：解析标题/URL/摘要，渲染为干净列表，禁止原样输出 `**[title](url)** — snippet` 格式

**1c. 产品定位确认（防止分类错误）**
- 通读 README 后回答：给谁用？解决什么问题？什么产品类别？
- ⚠️ **严禁仅凭关键词推断类别**
- README 不够清楚时，`web_fetch` 官网 docs 页面确认

**1d. 项目官网** → 产品定位、定价、团队页面、用户案例

**1e. 创始团队（必须搜索）**
- `web_search` "{创始人名} {公司名}" → LinkedIn、个人网站
- 从 GitHub 贡献者列表识别核心维护者，搜索其背景
- ⚠️ **不搜索团队 = 团队评分上限 4.0**

**1f. 融资信息**
- `web_search` "{公司名} funding valuation investors" → Crunchbase / PitchBook

**1g. 竞品搜索（必须基于真实产品类别）**
- 基于 1c 确认的产品定位搜索同类产品
- ⚠️ **竞品必须和项目解决同一个问题**
- `web_search` "{真实产品类别} alternatives competitors funding"
- **对每个竞品搜索 GitHub URL**：`web_search` "{竞品名} github"`
- 闭源产品记录官网 URL 代替

**1h. 市场数据**
- `web_search` "{准确细分市场名} market size TAM 2025 2026"

**1i. 媒体与热度**
- `web_search` "{项目名}" → 近 3 个月科技新闻报道
- `web_search` "{项目名} site:news.ycombinator.com" → HN 热度
- `web_search` "{项目名} OR {公司名} review" → Reddit、博客、Twitter/X
- 参考 Google Trends 判断搜索热度趋势

**1j. 市场信号增强（尽力获取）**
- `web_search` "{项目名} pypi downloads" 或 "{项目名} npm downloads" → 包下载量
- `web_search` "{项目名} used by dependents" → 被依赖数量
- `web_search` "{公司名} hiring jobs" → 招聘信号
- `web_search` "{项目名} enterprise customers case study" → 企业客户披露

### Step 2：数据校验（写报告前的强制检查）

| 必须确认 | 来源 | 未确认时处理 |
|---------|------|-------------|
| Stars 精确数字 | GitHub 页面 | 不写报告 |
| License 类型 | GitHub 页面 | 标注"未确认" |
| 产品真实类别 | README + 官网 | 不写报告 |
| 仓库所有者类型 | GitHub 页面 | 判断个人/组织 |
| 组织官网（若属于组织） | GitHub org 页面 | 必须找到并 web_fetch |
| 组织融资信息（若属于组织） | 搜索结果 | 标注"⚠️ 未发现" |
| 组织收入/客户信号（若属于组织） | 搜索结果 | 标注"⚠️ 未公开" |
| 项目在公司中的角色（若属于组织） | README + 官网 | 必须明确判断 |
| 创始人/核心团队至少 1 人 | 搜索结果 | 团队评分上限 4.0 |
| 至少 2 个真实竞品 | 搜索结果 | 标注"⚠️ 信息不足" |
| 至少 1 条媒体/社区信号 | 搜索结果 | 媒体评分上限 5.0 |
| 主维护者信息至少 1 条 | GitHub 页面/搜索 | 标注"⚠️ 待抓取" |

### Step 3：三层评分

基于校验后的数据，按三层评分体系打分。每个评分必须附数据支撑理由。

### Step 4：撰写报告

按模板输出，Markdown + HTML。

### Step 5：验收自检

- [ ] Investment Thesis 存在且在报告第一段
- [ ] 三层评分均有数据支撑
- [ ] Maintainer Risk Matrix 有明细和结论
- [ ] 融资表有来源列
- [ ] 战略动态为干净列表（无 markdown 噪声）
- [ ] 竞品表有可点击链接
- [ ] HTML 与 Markdown 信息一致

## 四、三层评分系统

### 总分计算

```
总分 = Health × 0.30 + Commercial × 0.30 + Structural × 0.30 + Momentum × 0.10
```

### 总分等级

| 等级 | 分数区间 | 含义 | 投资建议 |
|------|---------|------|---------|
| **S** | 9.0-10.0 | 极度看好 | 强烈推荐投资 |
| **A** | 7.5-8.9 | 非常看好 | 推荐投资 |
| **B** | 6.0-7.4 | 值得关注 | 条件性投资，需进一步验证 |
| **C** | 4.0-5.9 | 一般 | 暂不推荐，持续观察 |
| **D** | 0-3.9 | 不看好 | 不推荐 |

---

### Layer 1：🏥 Health Score（项目健康度）0-10

衡量项目本身的工程质量和社区活力。

| 子维度 | 权重 | 评估内容 | 数据来源 |
|--------|------|---------|---------|
| Star 增速 | 15% | 近 3 月 / 6 月增速，增长曲线形状 | GitHub 页面 / star-history |
| Issue 健康度 | 15% | Open vs Closed 比率、响应时间、Issue 质量 | GitHub Issues 页面 |
| PR 合并效率 | 15% | Open vs Merged PR 比率、合并周期 | GitHub PRs 页面 |
| Release 频率 | 15% | 近 6 月 release 次数、是否有规律 | GitHub Releases |
| Bus Factor | 25% | Top1 维护者贡献占比（详见 Maintainer Risk Matrix） | Contributors 页面 |
| 贡献者增长 | 15% | 外部贡献者占比、新贡献者增速 | Contributors 页面 |

```
9-10: Star 持续高增长 + Issue 响应 <48h + 规律 Release + Bus Factor <30% + 外部贡献 >30%
7-8:  多数指标健康，1-2 项一般
5-6:  增长放缓或 Bus Factor 偏高，但核心维护正常
3-4:  多项指标不健康（停更、Issue 积压、单人维护）
0-2:  项目濒临废弃或严重不健康
```

---

### Layer 2：💰 Commercial Potential Score（商业潜力）0-10

衡量从开源到商业化的转化能力。

| 子维度 | 权重 | 评估内容 |
|--------|------|---------|
| API/SaaS 化能力 | 20% | 产品是否天然适合 API 化或 SaaS 化？是否有托管版需求？ |
| 商业模式清晰度 | 20% | 是否已有定价页？Open Core / SaaS / Managed / Dual-license？ |
| 企业客户信号 | 20% | 是否有已知企业客户？是否有 Enterprise 版？ |
| 付费信号 | 20% | 是否有收入数据？ARR？付费用户数？ |
| 生态中心潜质 | 20% | 是否有插件/集成生态？是否有成为平台的潜力？ |

```
9-10: 已有规模收入（ARR >$1M）+ 清晰 Open Core + 企业客户 + 平台生态
7-8:  商业模式清晰 + 有早期付费客户 + 定价页存在
5-6:  有商业化意图（Pro 版/Enterprise 版），但未验证
3-4:  商业模式模糊，纯开源无商业化迹象
0-2:  无任何商业化意图或可能性
```

**商业路径图**（必须输出）：

```
当前位置 → 下一步变现路径 → 终局商业模式
例：OSS Library → Hosted API → Full Platform（类 Stripe 路径）
例：OSS Tool → Enterprise Edition → Vertical SaaS（类 GitLab 路径）
```

---

### Layer 3：🔮 Structural Opportunity Score（结构性机会）0-10

衡量宏观趋势是否利好该项目。

| 子维度 | 权重 | 评估内容 |
|--------|------|---------|
| 技术拐点 | 30% | 是否处于技术范式转换的浪尖？（如 AI 浪潮、云原生、WebAssembly） |
| 平台级机会 | 25% | 是否有成为基础设施/平台的可能？TAM 是否 >$10B？ |
| 监管/合规壁垒 | 20% | 是否有合规需求创造的结构性需求？（如数据隐私、AI 安全） |
| 地缘/产业迁移 | 15% | 是否有国产替代/技术主权/供应链迁移的机会？ |
| 时机判断 | 10% | 是否是"为什么是现在？"的最佳回答 |

```
9-10: 多个结构性利好叠加（技术拐点 + 平台机会 + 合规需求），时机完美
7-8:  有 1-2 个明确的结构性驱动力
5-6:  赛道稳定增长，但无突破性拐点
3-4:  市场成熟，增长空间有限
0-2:  市场萎缩或无结构性支撑
```

---

### Community Momentum（社区动能）0-10

独立维度，低权重（10%），但作为信号验证。

| 子维度 | 评估内容 |
|--------|---------|
| 媒体报道 | Tier-1/Tier-2/社区来源覆盖度 |
| HN/Reddit 热度 | 讨论频率、评论数、情感倾向 |
| Google Trends | 搜索热度趋势 |
| 包下载量 | npm / PyPI / Docker pulls |
| 被依赖数量 | dependents / used-by 数据 |
| 招聘信号 | 公司是否在招聘相关岗位 |

```
9-10: 全方位火爆（Tier-1 密集报道 + HN 首页 + 下载量激增 + 大量依赖）
7-8:  多数信号正向
5-6:  有一定关注但不突出
3-4:  低调运行，外部信号弱
0-2:  几乎无外部讨论
```

**⚠️ 热度 vs 实质判断（必须输出）：**
- 高热度 + 高 Health = ✅ 实力型增长
- 高热度 + 低 Health = ⚠️ 炒作泡沫风险
- 低热度 + 高 Health = 🔍 低调实力派（潜在价值洼地）
- 低热度 + 低 Health = ❌ 不推荐

## 五、Maintainer Risk Matrix（v5 升级版）

### 必须抓取的数据

| 数据项 | 来源 | 方法 |
|--------|------|------|
| Top 维护者列表（近 90 天） | GitHub Contributors 页面 | 提取 login / commits / share |
| Top1 贡献占比 | 上述数据计算 | `top1_commits / total_commits_90d` |
| 创始人身份 | web_search | 判断 top 维护者是否即创始人 |

### 六维风险矩阵

| 风险维度 | 评估内容 | 判定标准 |
|---------|---------|---------|
| 🧑‍💻 维护者集中度 | Top1 贡献占比 | ≥50% 高风险 · 30-50% 中风险 · <30% 低风险 |
| 👤 创始人依赖 | 创始人是否 = 唯一核心维护者 | 是 = 高风险 · 否 = 低风险 |
| 🔗 API/上游依赖 | 是否重度依赖单一 API/服务（如 OpenAI API） | 单一依赖 = 高 · 多供应商 = 低 |
| 🏢 公司化风险 | 项目是否过度依赖单一公司资金/决策 | 公司控制 + 无社区治理 = 高 |
| 📜 许可证风险 | 许可证是否可能引发分叉/采用障碍 | AGPL/BSL = 中 · MIT/Apache = 低 |
| 🌍 社区梯队深度 | 除 Top1 外是否有 ≥2 名活跃维护者 | <2 人 = 高 · 2-5 人 = 中 · >5 人 = 低 |

### 输出格式

```markdown
### Maintainer Risk Matrix

| 风险维度 | 状态 | 风险等级 |
|---------|------|---------|
| 🧑‍💻 维护者集中度 | Top1: {login} ({X}%) | 🔴/🟡/🟢 |
| 👤 创始人依赖 | {是/否} | 🔴/🟢 |
| 🔗 API/上游依赖 | {描述} | 🔴/🟡/🟢 |
| 🏢 公司化风险 | {描述} | 🔴/🟡/🟢 |
| 📜 许可证风险 | {许可证类型} | 🔴/🟡/🟢 |
| 🌍 社区梯队深度 | {活跃维护者数} | 🔴/🟡/🟢 |

**综合风险等级：{低/中/高}**
**专项说明**：{一段话结论}
```

## 六、投资阶段判断

### 项目阶段识别

| 阶段 | 特征 | 典型指标 |
|------|------|---------|
| 💡 Idea | 仅有仓库，无社区 | Stars <100, 无外部贡献者 |
| 🌱 Early OSS | 有早期社区，功能基本可用 | Stars 100-2k, 有 Issues/PRs |
| 📈 PMF Signal | 增长加速，有采用信号 | Stars 2k-20k, 企业使用迹象 |
| 💼 Commercializing | 开始商业化探索 | 有定价页/Enterprise版, 早期收入 |
| 🚀 Scaling | 规模化增长 | ARR >$1M, 大量企业客户 |

### 投资窗口判断

| 投资窗口 | 对应项目阶段 | 进入时机信号 |
|---------|------------|------------|
| Pre-seed | Idea → Early OSS | 创始人极强 + 赛道明确 |
| Seed | Early OSS → PMF Signal | Star 加速 + 首批企业用户 |
| Series A | PMF Signal → Commercializing | 商业模式验证 + ARR 增长 |
| 战略投资 | Commercializing → Scaling | 生态协同价值 > 财务回报 |
| 观察 | 任何阶段 | 信号不足，需更多数据 |

## 🎯 Investment Thesis

> **核心判断（一句话）**：{为什么应该/不应该投资}

**三条逻辑支撑**：
1. {逻辑 1 — 最强理由}
2. {逻辑 2}

## 九、分析原则

### 1. 数据驱动，零容忍虚假数据
Stars/Forks/Issues 等必须来自 web_fetch 实际页面。绝不编造。

### 2. Investment Thesis First
报告第一段就给出核心判断。读者应在 30 秒内知道结论。

### 3. 产品定位先行，拒绝关键词分类
必须通读 README 和官网后再做分类。常见错误：
- ❌ 含"agent" → agent 框架 → 竞品列 LangChain
- ✅ 通读后发现是个人 AI 助手 → `personal-assistant`

### 4. 评分要有区分度
同一报告最高分和最低分之差应 ≥ 3 分。允许 3 分和 9 分并存。

### 5. 竞品必须精准
- 竞品必须解决同一问题
- 每个竞品有搜索验证的数据
- 每个竞品有可点击链接（GitHub URL 优先）

### 6. 融资记录必须有来源
每条融资记录附来源链接。无来源显示 `—`。

### 7. 战略动态必须清洗
禁止原样输出 `**[title](url)**` 格式。必须解析为结构化列表后渲染。

### 8. Maintainer 风险不可跳过
必须有主维护者列表 + 六维风险矩阵 + 结论。

### 9. 客观坦诚
竞争对手优势如实承认。风险分析具体到公司名和数字。

| 风险评估全是模板语言 | 具体到公司名、金额、事件 |
| Star 暴增就给满分 | 检查 Fork 率、Issue 质量、下载量 |

## 十二、容错与降级策略

| 故障场景 | 降级方案 |
|---------|---------|
| GitHub API 403 限流 | 复用 web_fetch 页面数据；标注"数据来源：页面抓取" |
| Contributors 数据缺失 | 从 GitHub Contributors 页面提取可见数据；标注"⚠️ 维护者数据不完整" |
| 融资信息搜索无果 | 标注"⚠️ 未发现公开融资信息"；不编造数据 |
| 官网无法访问 | 标注"⚠️ 官网不可达"；仅基于 GitHub 和搜索结果分析 |
| 竞品搜索结果不佳 | 分析相邻赛道潜在威胁；标注"⚠️ 竞品信息不足" |
| 市场数据缺失 | 基于搜索结果做定性判断；标注"⚠️ 市场数据待验证" |
